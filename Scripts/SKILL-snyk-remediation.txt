# Snyk Vulnerability Remediation Skill

## Purpose

This file instructs GitHub Copilot Chat on how to remediate Snyk-reported
dependency vulnerabilities in Citi CHUB Spring Boot microservices. When you
ask Copilot to fix vulnerabilities, it reads this file and applies the
correct fix strategy.

## Scope

- Maven pom.xml dependency version upgrades
- JDK 17 to 21 migration (properties + compiler plugin)
- Detect and fix version mismatches (e.g., Boot 4.x deps in Boot 3.x projects)
- Does NOT modify Java source code
- Does NOT add or remove dependencies (unless flagging a mismatch)

## How to Invoke

Open this repo in VS Code and use any of these prompts in Copilot Chat:

```
@workspace Fix Snyk vulnerabilities
@workspace Upgrade dependencies to secure versions
@workspace Apply CVE remediation
@workspace Migrate to JDK 21
@workspace Check for version mismatches
```

## Project Structure (Citi CHUB Microservices)

These microservices use a **custom Citi parent BOM**, NOT the standard
Spring Boot parent. This is critical for the fix strategy.

```xml
<parent>
    <groupId>com.citi.169073.ccp.rel</groupId>
    <artifactId>ccp-spring-boot-starter-parent-master</artifactId>
    <version>2026.6.13.RELEASE</version>
</parent>
```

### Key POM Characteristics:
- Parent BOM: `ccp-spring-boot-starter-parent-master` (Citi's internal BOM)
- Packaging: WAR (deployed to external Tomcat)
- Most Spring/Boot dependency versions are inherited from the parent BOM
- Some dependencies have explicit versions in the POM
- JDK version is set in `<properties>` AND in `maven-compiler-plugin` config
- Tomcat is excluded from spring-boot-starter-web, then re-added as provided
- **CONFIRMED:** The Citi parent BOM DOES respect standard Spring Boot property
  overrides (spring-framework.version, spring-retry.version, micrometer.version, etc.)

## Fix Strategy (Priority Order)

### Priority 1: Add Version Properties (PROVEN APPROACH)

The Citi parent BOM (`ccp-spring-boot-starter-parent-master`) respects standard
Spring Boot version property names. Adding these properties in `<properties>`
overrides the transitive dependency versions at build time.

Add these to the `<properties>` section of pom.xml:

```xml
<properties>
    <!-- JDK upgrade -->
    <java.version>21</java.version>
    <maven.compiler.target>21</maven.compiler.target>
    <maven.compiler.source>21</maven.compiler.source>

    <!-- Snyk CVE remediation: version overrides -->
    <spring-framework.version>6.2.19</spring-framework.version>
    <spring-retry.version>2.0.13</spring-retry.version>
    <micrometer.version>1.15.12</micrometer.version>
    <netty.version>4.1.135.Final</netty.version>
    <tomcat.version>10.1.55</tomcat.version>

    <!-- ... existing properties remain unchanged ... -->
</properties>
```

**Also update the maven-compiler-plugin** (JDK version is hardcoded there too):
```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <configuration>
        <source>21</source>
        <target>21</target>
    </configuration>
</plugin>
```

### Priority 2: Add dependencyManagement (ONLY if Priority 1 doesn't work)

old versions, then the parent BOM doesn't respect that specific property name.

### Priority 3: Fix Version Mismatches

**IMPORTANT:** Check for dependencies where someone hardcoded a version from a
different Spring Boot major version. This causes class conflicts and build failures.

Common mismatch pattern to detect and fix:
```xml
<!-- BAD: Boot 4.x dependency in a Boot 3.x project -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-webmvc-test</artifactId>
    <version>4.0.6</version>  <!-- WRONG for Boot 3.x project! -->
    <scope>test</scope>
</dependency>

<!-- FIXED: Remove explicit version, let parent BOM manage it -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-webmvc-test</artifactId>
    <scope>test</scope>
</dependency>
```

**Rule:** For any `org.springframework.boot` or `org.springframework.cloud`
dependency, NEVER hardcode a version. Let the parent BOM manage it. If you
see a hardcoded version, remove it.

Exceptions (these DO need explicit versions because they're not in the parent BOM):
- `cmr-em-microserviceframework` (internal Citi library)
- `javapasswordsdk` (CyberArk)
- `springdoc-openapi-ui` (not managed by Spring Boot BOM)
- `hsqldb`, `ojdbc8` (database drivers with specific classifier/versions)
- `commons-collections4`, `commons-io`, `org.json:json` (standalone utilities)

## JDK 17 to 21 Migration (5 Places to Update)

JDK version must be changed in ALL of these locations:

| Location | Before | After |
|---|---|---|
| `<properties><java.version>` | 17 | 21 |
| `<properties><maven.compiler.target>` | 17 | 21 |
| `<properties><maven.compiler.source>` | 17 | 21 |
| maven-compiler-plugin `<source>` | 17 | 21 |
| maven-compiler-plugin `<target>` | 17 | 21 |

---

## Validation Checklist (After Applying Fixes)

Run these commands to confirm the fix worked:

```bash
# 1. Build should pass
mvn clean compile -DskipTests

# 2. Verify Spring Framework version (should show 6.2.19)
mvn dependency:tree -Dincludes=org.springframework:spring-core

# 3. Verify Spring Retry version (should show 2.0.13)
mvn dependency:tree -Dincludes=org.springframework.retry:spring-retry

# 4. Verify Micrometer version (should show 1.15.12)
mvn dependency:tree -Dincludes=io.micrometer:micrometer-core

# 5. Check for Boot version mismatches (should return nothing for Boot 4.x in Boot 3.x project)
mvn dependency:tree | findstr "4.0"
```

## Maintenance

When new Snyk findings appear:
1. Check if a newer `ccp-spring-boot-starter-parent-master` is available on Nexus
2. If yes, update parent version — this is the cleanest fix
3. If no, add/update version properties in `<properties>` (Priority 1)
4. If properties don't resolve the dep, use `<dependencyManagement>` (Priority 2)
5. Always scan for version mismatches (Boot 4.x deps in Boot 3.x projects)
6. Update this skill file with new target versions
